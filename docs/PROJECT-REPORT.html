01الملخّص التنفيذي (Abstract)

نظام إدارة قاعدة بيانات شامل وآمن لمستشفى العالمية IVF مبنياً على ثلاث طبقات أمان متزامنة: RBAC على مستوى التطبيق (7 أدوار، 68 صلاحية)، RLS من قاعدة البيانات نفسها (21 سياسة)، وتدقيق append-only مع تقنيع البيانات. يستخدم PostgreSQL 15، Node.js/Express، وواجهة عربية RTL. يتضمّن 22 جدولاً معايّراً، 30+ endpoint، و61 اختبار ناجح مع CI/CD مستمر.

النطاق · Scope

هذا التقرير دراسة هندسية محدّدة لنظام قاعدة بيانات مستشفى واحد مع نموذج أمان ثلاثي الطبقات، وليس مسحاً عاماً لأنظمة قواعد البيانات. كل رقم وكل سياسة موثّقة من شيفرة المشروع الفعلية.

02المقدمة والخلفية

المؤسسات الطبية تحتفظ ببيانات حسّاسة جداً (سجلات طبية، بيانات شخصية، بيانات مالية). الأنظمة التقليدية تعاني من: ضعف التحكم بالوصول (لا تمييز دقيق)، غياب أمان مستوى الصفّ (إذا اخترق المهاجم قاعدة البيانات يرى كل شيء)، عدم وجود سجلّ تدقيق شامل، وإفشاء البيانات الحسّاسة.

هذا المشروع يقدّم حلاً متكاملاً:

  • RBAC متقدم: 7 أدوار و68 صلاحية بـ least privilege
  • Row-Level Security: 21 سياسة تفرضها قاعدة البيانات
  • Audit Logging Append-only: لا يسمح بالتعديل
  • Data Masking: إخفاء أرقام حسّاسة حسب الدور
  • التشفير الآمن: bcrypt 12 جولة، JWT موقّعة

04المتطلّبات

المتطلّبات الوظيفية الأساسية

REQ-F1: إدارة المرضى — تسجيل، MRN تلقائي، RLS حسب الدور.

REQ-F2: المواعيد — حجز، منع تضارب (UNIQUE)، RLS حسب المريض والطبيب.

REQ-F3: السجلات الطبية — إنشاء، تحديث العلامات، append-only (لا حذف).

REQ-F4: التشخيصات — إضافة بـ ICD-10، تصنيف، رؤية محدودة.

REQ-F5: الأدوية والوصفات — إنشاء، صرف، إدارة مخزون.

REQ-F6: المختبر — طلب فحص، إدخال نتائج، RLS للنتائج.

REQ-F7: الفوترة — إصدار فاتورة، دفع على دفعات، لا حذف (تحديث فقط).

REQ-F8: الموظفون — إضافة، تعيين أقسام، رؤية بيانات محدودة.

REQ-F9: المصادقة — bcrypt آمن، JWT، قفل بعد 5 محاولات.

REQ-F10: التدقيق — تسجيل كل عملية (من، ماذا، متى، IP).

المتطلّبات غير الوظيفية

الأمان: RBAC (7 أدوار، 68 صلاحية)، RLS (21 سياسة)، bcrypt، JWT، قفل الحسابات، تدقيق append-only، تقنيع البيانات، OWASP (SQL injection, XSS, CSRF), Rate Limiting (100/دقيقة عام، 10/دقيقة دخول), Helmet headers.

الأداء: < 100ms للطلبات الشائعة، < 500ms للبحث.

الموثوقية: FK، UNIQUE، CHECK، NOT NULL، نسخ احتياطية دورية.

الجودة: اختبارات > 80% coverage، CI/CD GitHub Actions، توثيق شامل (ERD، RBAC-MATRIX، Setup Guide).

  • RBAC (4 جداول): roles (7)، permissions (68)، role_permissions، user_roles
  • المنظمة (4): departments، staff، doctors، nurses
  • المرضى (2): patients، users
  • الخدمات السريرية (7): appointments، medical_records، diagnoses، prescriptions، prescription_items، lab_tests، lab_orders

التطبيع والعلاقات

3NF/BCNF: لا تكرار بيانات، فصل الجداول حسب الموضوع. مثال: جدول staff منفصل عن doctors/nurses.

العلاقات:

  • 1:1: doctors/nurses من staff (UNIQUE)
  • 1:N: 16 علاقة (مثل: 1 مريض → N سجل طبي)
  • N:M: 3 علاقات (role_permissions, user_roles, prescription_items)

قيود السلامة

-- CHECK: فصيلة الدم من 8 خيارات
blood_type VARCHAR(3) CHECK (blood_type IN ('A+','A-','B+','B-','AB+','AB-','O+','O-'))

-- CHECK: نوع الموظّف من 6 خيارات
staff_type VARCHAR(30) CHECK (staff_type IN ('doctor','nurse','receptionist',...)

-- UNIQUE: لا موعدان للطبيب في نفس الوقت
UNIQUE (doctor_id, scheduled_at)

الفهارس والأداء

28+ فهرس موضوعة بذكاء:

  • فهارس المفاتيح الأجنبية (سرعة JOIN)
  • فهارس جزئية (partial indexes) للبيانات الفعالة
  • فهارس مركّبة (composite) للاستعلامات الشائعة

مثال:

CREATE INDEX idx_appointments_doctor_date ON appointments(doctor_id, scheduled_at);

        ↓ HTTP/HTTPS

        ↓ SQL + RLS

طبقة العرض (Frontend)

الملفات: index.html (دخول)، dashboard.html (لوحة حسب الدور)، css/style.css (RTL)، js/api.js (HTTP wrapper)، js/auth.js (المصادقة)، js/dashboard.js (العمليات).

DIAGRAM 02Three-tier architecture — secure request flowالمعمارية ثلاثية الطبقات وتدفّق الطلب الآمن

7 واجهات مختلفة حسب الدور: Admin، Doctor، Nurse، Receptionist، Lab Technician، Pharmacist، Patient.

طبقة التطبيق (Backend)

الهيكل:

  • server.js: نقطة الدخول
  • app.js: إعداد Express + Middleware
  • config/db.js: PostgreSQL Pool + withUserContext (تفعيل RLS)
  • middleware/: auth (JWT)، rbac (صلاحيات)
  • controllers/: منطق الأعمال (9 controllers)
  • routes/: تعريف 30+ endpoint
  • utils/: AppError، asyncHandler

الأمان:

  • JWT: 15 دقيقة مع roles وpermissions
  • RBAC: requireRole، requirePermission، requireAnyPermission
  • Helmet: رؤوس HTTP أمنية
  • CORS: محصور برأصل واحد
  • Rate Limiting: 100/دقيقة عام، 10/دقيقة login

تدفق الطلب

1. Frontend يرسل JWT في Authorization header

2. authenticateToken: يفك JWT ويملأ req.user

3. requirePermission: يفحص الصلاحيات

4. Controller: يستدعي withUserContext(userId, roles, SQL)

5. PostgreSQL: يفعّل RLS تلقائياً (SET LOCAL)، ينفذ الاستعلام، ينطلق audit trigger

6. Response: يرد 200/403/404 مع البيانات أو الخطأ

طبقة البيانات

RLS تطبيق: معاملات منفصلة، SET LOCAL app.current_user_id، تطبيق سياسات تلقائياً.

app_role: دور التطبيق بـ NOBYPASSRLS لفرض RLS حتى عند الاختراق.

Audit Triggers: 22 trigger على الجداول الحسّاسة تسجل كل INSERT/UPDATE/DELETE.

07نموذج RBAC و RLS

RBAC — 7 أدوار، 68 صلاحية

الأدوار:

المبدأ · Principle

RBAC يقرّر ما يُسمح فعله، وRLS يقرّر أي صفوف تُرى. الطبقتان متكاملتان لا بديلتان؛ فلو سقطت طبقة التطبيق تبقى سياسات RLS تحرس البيانات على مستوى المحرّك نفسه.

DIAGRAM 03Simplified RBAC matrix — roles × resourcesمصفوفة RBAC المبسّطة: الأدوار مقابل الموارد

1. Admin: 68 صلاحية (كاملة)

2. Doctor: ~25 صلاحية (سجلات مرضاه، تشخيصات، وصفات، طلب فحوصات)

3. Nurse: ~15 صلاحية (قراءة سجلات، تحديث علامات حيوية، تصحيح المواعيد)

4. Receptionist: ~20 صلاحية (تسجيل مرضى، حجز مواعيد، فوترة)

5. Lab Technician: ~5 صلاحيات (قراءة طلبات، إدخال نتائج)

6. Pharmacist: ~8 صلاحيات (قراءة وصفات، صرف، إدارة مخزون)

7. Patient: ~10 صلاحيات (ملفه الخاص، مواعيده، فواتيره)

الصيغة: resource:action (مثل: patients:read, medical_records:create, prescriptions:dispense).

RLS — 21 سياسة على 7 جداول

الجداول المحمية:

  • patients (3 سياسات): SELECT، INSERT، UPDATE حسب الدور والـ ID
  • medical_records (3 سياسات): الطبيب يرى مرضاه، الممرّض يرى القسم، لا حذف
  • appointments (3 سياسات): المريض مواعيده، الطبيب مواعيده، الاستقبال الكل
  • diagnoses (3 سياسات): الطبيب يرى تشخيصاته، المريض يرى تشخيصاته
  • prescriptions (3 سياسات): الطبيب ينشئ، الصيدلي يصرف، المريض يرى
  • lab_orders (3 سياسات): الطبيب ينشئ، الفنّي ينفذ، النتائج محدودة
  • invoices (3 سياسات): المريض يرى فواتيره، الاستقبال الكل، لا تعديل

دوال المساعدة:

app_current_user_id()      -- معرّف المستخدم
app_has_role(role_text)    -- هل يملك الدور؟
app_current_doctor_id()    -- معرّف الطبيب
app_current_patient_id()   -- معرّف المريض

مثال عملي

-- Doctor يرى مرضاه فقط
SET LOCAL app.current_user_id = 5;    -- الطبيب
SET LOCAL app.current_role = 'doctor';
SELECT * FROM patients;  -- يرى فقط مرضاه عبر medical_records

-- Nurse في قسم يرى سجلات ذاك القسم
SET LOCAL app.current_user_id = 6;
SET LOCAL app.current_role = 'nurse';
SELECT * FROM medical_records;  -- فقط سجلات القسم

-- Patient يرى ملفه فقط
SET LOCAL app.current_user_id = 7;
SET LOCAL app.current_role = 'patient';
SELECT * FROM patients;  -- فقط ملفه (id = app_current_patient_id())

الدفاع المتعمّق · Defense in Depth

لا توجد طبقة أمان واحدة كافية. يجمع المشروع المصادقة + التفويض + عزل الصفوف + التدقيق + التقنيع بحيث يتطلّب اختراق البيانات تجاوز أربع طبقات مستقلّة معاً.

const hash = await bcrypt.hash('Passw0rd!', 12);
// يُخزَّن: $2b$12$R9h.cIPz0gi.URNNHV3/2OPST9/PgBkqquzi.Ss7KIUgO2t0jKMUm
await bcrypt.compare('Passw0rd!', hash);  // true

JWT: توكنات موقّعة 15 دقيقة تتضمّن roles وpermissions:

{
  "id": 5,
  "roles": ["doctor"],
  "permissions": ["patients:read", "medical_records:create", ...],
  "iat": 1719767400,
  "exp": 1719768300
}

التدقيق (Audit Logging)

جدول audit_logs: append-only، يسجل:

  • من: user_id
  • ماذا: INSERT/UPDATE/DELETE
  • أين: entity_type, entity_id
  • الفوارق: old_values, new_values (JSONB)
  • متى: created_at، ip_address

إجبار Append-Only: Trigger يرفع EXCEPTION عند محاولة UPDATE/DELETE، + REVOKE من app_role.

مثال:

{
  "user_id": 5,
  "action": "UPDATE",
  "entity_type": "medical_records",
  "old_values": {"temp": 37.5, "pulse": 72},
  "new_values": {"temp": 39.0, "pulse": 95},
  "ip_address": "192.168.1.100",
  "created_at": "2026-06-30T14:32:15Z"
}

تقنيع البيانات (Data Masking)

دوال التقنيع:

mask_phone('07901234567')        → '*******4567'
mask_national_id('123456789012') → '**********12'
mask_email('ahmed@hospital.iq')  → 'a***@hospital.iq'

Views حسب الدور:

  • Receptionist: رقم وطني وهاتف مُقنّع
  • Doctor: بيانات كاملة (مخوّل سريرياً)
  • Nurse: بيانات سريرية بدون معرّفات
  • Pharmacist: معلومات الصرف الأساسية فقط

الدفاع الإضافي

Helmet: رؤوس أمنية (CSP, X-Frame-Options, HSTS).

CORS: محصور برأصل واحد (بدون *).

Rate Limiting: 100/دقيقة عام، 10/دقيقة login.

قفل الحساب: بعد 5 محاولات فاشلة (15 دقيقة).

معالجة الأخطاء: SQL error codes مثل 23505 (duplicate) و23514 (check violation).

11التحديات والحلول

7 مشاكل واقعية وحلولها

1. Duplicate Roles

المشكلة: INSERT متكرّر على جدول roles يرفع constraint violation.

الحل: ON CONFLICT (name) DO NOTHING لـ idempotency.

التأثير: ملفات قاعدة البيانات آمنة للتشغيل المتكرّر.

2. Composite Key ID Extraction

المشكلة: جدول user_roles بدون عمود id، trigger محاولة استخراج NEW.id يفشل.

الحل: استخدام (v_new ->> 'id')::BIGINT بدل NEW.id.

التأثير: دعم audit logging لأي جدول.

3. NULL in BIGINT Cast

المشكلة: current_setting('app.current_user_id')::BIGINT عندما تكون فارغة ترفع خطأ.

الحل: NULLIF(current_setting(...), '')::BIGINT.

التأثير: معالجة آمنة للبيانات الفارغة.

4. RLS Bypass (Superuser)

المشكلة: postgres superuser يتجاوز RLS تلقائياً.

الحل: SET LOCAL ROLE app_role (NOBYPASSRLS) قبل كل استعلام.

التأثير: فرض RLS حتى عند الاختراق.

5. Permission Logic Flaws

المشكلة: Nurse لا ترى سجلات (bيانات ward ناقصة في جدول nurses).

الحل: ملء seed data بشكل كامل، تحسين SQL queries.

التأثير: بيانات موثوقة وسياسات دقيقة.

6. JWT Claims Accuracy

المشكلة: Pharmacist لا تملك permissions في JWT رغم وجودها في DB.

الحل: استعلام صحيح يجلب كل الصلاحيات من role_permissions.

التأثير: صلاحيات دقيقة في كل طلب.

7. Error Handling

المشكلة: رد غير واضح عند UNIQUE violation (PostgreSQL error 23505).

الحل: معالج خاص لـ SQL error codes (23505 duplicate، 23514 check violation).

التأثير: رسائل خطأ واضحة للمستخدم.
